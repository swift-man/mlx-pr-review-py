# AGENTS.md

이 문서는 이 저장소에서 작업하는 사람과 에이전트를 위한 공통 작업 기준입니다.
목표는 "돌아가는 코드"를 넘어서, 운영 중에도 추적 가능하고 안전하게 고칠 수 있는 구조를 유지하는 것입니다.

## 1. 프로젝트 이해

이 저장소는 GitHub webhook을 받아 PR diff를 수집하고, MLX 기반 리뷰 결과를 만든 뒤 GitHub Review API에 등록하는 서버입니다.

핵심 경로는 아래와 같습니다.

- `review_runner/webhook_app.py`: FastAPI 진입점
- `review_runner/review_service.py`: GitHub API 호출, 프롬프트 생성, 리뷰 등록
- `review_runner/mlx_review_client.py`: MLX 모델 호출과 응답 정규화
- `scripts/install_local_review.sh`: 실행 복사본 배포
- `scripts/run_webhook_server.sh`: 실행 복사본에서 서버 시작

## 2. 작업 원칙

모든 변경은 아래 원칙을 기본값으로 삼습니다.

- 사용자에게 보이는 동작을 먼저 이해한 뒤 수정합니다.
- 문제를 재현하거나 로그 흐름을 확인한 뒤 고칩니다.
- 작은 수정이라도 운영 경로에 영향이 있으면 실패 시나리오를 함께 점검합니다.
- 문서, 스크립트, 코드가 서로 다른 말을 하지 않도록 함께 맞춥니다.

## 3. SOLID 원칙

이 저장소에서는 SOLID 원칙을 "무조건 추상화 늘리기"가 아니라, 변경 비용을 줄이는 기준으로 사용합니다.

### 단일 책임 원칙

- 한 함수는 한 가지 이유로만 바뀌게 유지합니다.
- `webhook_app.py`는 HTTP 입구에 집중하고, 리뷰 생성과 GitHub 등록 로직은 서비스 레이어에 둡니다.
- MLX 실행 로직과 GitHub API 로직은 같은 함수에서 뒤섞지 않습니다.

### 개방-폐쇄 원칙

- 새로운 리뷰 클라이언트나 인증 방식이 추가되더라도 기존 핵심 흐름을 크게 흔들지 않게 설계합니다.
- 조건문을 계속 누적하기보다, 기존 함수 경계를 유지한 채 확장 지점을 분리합니다.

### 리스코프 치환 원칙

- mock 클라이언트, sample 클라이언트, 실제 MLX 클라이언트는 호출 계약을 동일하게 유지합니다.
- 대체 구현을 붙였을 때 호출부 수정이 필요하면 구조를 다시 봅니다.

### 인터페이스 분리 원칙

- 한 모듈이 필요하지 않은 설정값이나 반환 필드까지 억지로 알지 않게 합니다.
- 예를 들어 webhook 레이어는 모델 세부 옵션을 몰라도 동작해야 합니다.

### 의존성 역전 원칙

- 상위 정책은 하위 구현 세부사항에 직접 묶이지 않게 유지합니다.
- 가능하면 "리뷰를 생성한다", "리뷰를 등록한다" 같은 역할 중심 경계를 먼저 세우고, MLX나 GitHub REST 세부 구현은 아래 계층으로 둡니다.

## 4. 클린 아키텍처 도입 기준

가능한 범위에서 클린 아키텍처를 검토합니다. 다만 이 저장소는 비교적 작은 운영 서비스이므로, 계층을 늘리는 것 자체가 목적이 되면 안 됩니다.

우선 검토할 분리는 아래 순서입니다.

1. 엔트리포인트 계층
   `webhook_app.py`, CLI 진입점, 스크립트 연결부
2. 유스케이스 계층
   "PR 파일 수집", "리뷰 생성", "GitHub 리뷰 등록", "재시도/폴백 처리"
3. 인프라 계층
   GitHub API, MLX subprocess, 파일 시스템, 환경 변수

도입 기준은 아래와 같습니다.

- 같은 함수가 HTTP, 비즈니스 규칙, 외부 API 호출을 동시에 다루면 분리를 검토합니다.
- 테스트가 mock 남용 없이는 불가능하면 경계를 다시 봅니다.
- 운영 장애 대응 시 "어느 계층에서 실패했는지" 로그만 보고 구분하기 어렵다면 구조 개선을 검토합니다.

도입을 미루는 경우도 분명히 적어둡니다.

- 파일 수가 늘기만 하고 책임 분리가 실제로 선명해지지 않을 때
- 단일 변경에 여러 계층을 억지로 건드려야 할 때
- 현재 팀 규모와 서비스 복잡도에 비해 추상화 비용이 더 클 때

즉, 클린 아키텍처는 가능한 한 도입하되 과설계는 피합니다.

## 5. 코드 작성 기준

- 기본은 단순한 함수와 명확한 이름입니다.
- 예외 메시지는 운영 로그만 보고도 원인을 짐작할 수 있게 씁니다.
- 외부 API 실패, MLX 실패, dry-run 경로는 반드시 로그로 구분 가능해야 합니다.
- 환경 변수는 진입점 근처에서 해석하고, 핵심 로직에는 필요한 값만 전달합니다.
- 폴백 로직은 "언제 발동하는지"가 명확해야 하며, 조용히 숨겨진 동작이 되지 않게 로그를 남깁니다.

## 6. 테스트 기준

- 버그를 고쳤다면 같은 유형의 재발을 막는 테스트를 함께 추가합니다.
- subprocess, GitHub API, MLX 호출은 가능한 한 계약 중심으로 테스트합니다.
- 회귀 가능성이 큰 운영 경로는 happy path보다 실패 경로 테스트를 우선합니다.

최소한 아래 케이스를 자주 의식합니다.

- GitHub 인증 실패
- webhook 서명 실패
- MLX subprocess abort
- CPU fallback 동작
- dry-run에서 실제 POST가 생기지 않는지 확인

## 7. 문서와 운영 동기화

- 실행 경로가 원본 repo와 배포 복사본으로 나뉘어 있으므로, 코드 수정 후 재배포 여부를 항상 확인합니다.
- README, 스크립트 사용법, 실제 기본 경로가 다르면 문서도 같이 고칩니다.
- 운영자가 따라 하는 절차는 복붙 가능한 명령 형태로 남깁니다.

재실행이 필요한 상태라면 아래 명령을 안내합니다.

```bash
cd /Users/m4_25/develop/codereview
./scripts/redeploy_local_review.sh /Users/runner/pr-review
```

## 8. 변경 전 체크리스트

- 이 변경이 HTTP 입구, 유스케이스, 인프라 중 어디에 속하는지 설명 가능한가
- 함수 책임이 이전보다 더 선명해졌는가
- 새 분기가 생겼다면 테스트와 로그가 함께 추가되었는가
- README나 운영 절차 문서도 같이 바뀌어야 하는가

## 9. 변경 후 체크리스트

- 관련 테스트를 돌렸는가
- 실패 로그가 원인을 드러내는가
- 배포 복사본 기준 재시작 절차가 필요한가
- 재실행이 필요하다면 아래 명령을 함께 안내했는가

```bash
cd /Users/m4_25/develop/codereview
./scripts/redeploy_local_review.sh /Users/runner/pr-review
```

- 과한 추상화 없이도 다음 변경이 쉬워졌는가

이 문서는 "무조건 크게 리팩터링하라"는 뜻이 아닙니다.
작은 서비스일수록 더 중요합니다. 함수 하나, 로그 한 줄, 경계 하나를 제대로 두는 것. 그게 유지보수성을 만듭니다.
